# Communication Module

## Purpose
- Provide secure messaging, notifications, and broadcast communication.

## Core Data Model
- `messages`: direct messages between users.
- `notifications`: system alerts (deadlines, reminders).
- `broadcast_messages`: role‑based broadcast announcements.

## Architecture and Data Flow
- Users send messages to other roles based on RLS rules.
- Notifications are generated by system events or admins.
- Broadcast messages target roles or classes.

## API Surface (Representative)
- `GET /api/messages`
- `GET /api/messages/sent`
- `POST /api/messages`
- `PUT /api/messages/:id` (mark read)
- `DELETE /api/messages/:id`
- `GET /api/notifications`
- `POST /api/notifications`
- `POST /api/broadcast_messages`

## UI Structure
- Inbox and Sent tabs.
- Unread badge indicators.
- Compose modal with recipient filters.

## Security and RBAC
- Users only access their own inbox/sent.
- Teachers can message students and parents in assigned classes.
- Admins can broadcast to selected roles.

## Constraints and Rules
- Message visibility strictly enforced by RLS.
- Notifications should be immutable; use read flags instead of deletes.

## Improvement Ideas
- Real‑time updates via Supabase Realtime.
- Message templates and scheduled broadcasts.
